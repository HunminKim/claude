## 1. 【높음】 파일 쓰기/읽기 cp949 crash — ✅ 해결됨 (§0 참조)

> **해결**: 아래 분석대로 훅 전체 텍스트 I/O 에 `encoding="utf-8"`(읽기는 마이그레이션 안전을 위해
> `errors="ignore"`)을 전수 명시했다. 실제 누락은 본문이 센 9곳(쓰기)보다 넓어 **대칭 읽기**
> (update_docs read 4곳·`open`, plan_gate_lib/prompt_log_lib `load_*`, todo.md CLI read 3곳)까지
> 포함했다. AST 기반 smoke_test `[37]` 가 향후 누락을 컴파일 타임에 차단한다. 아래는 원 분석 기록.

### 현상
`Path.write_text(...)` 가 `encoding=` 미지정이면 **locale 기본(Windows=cp949)** 으로 파일을 쓴다.
내용에 `✅`(✅) 등 비-cp949 문자가 있으면 `UnicodeEncodeError` 로 훅·CLI 가 죽는다.
§0 의 stdout reconfigure 는 **출력 스트림만** 덮으므로 이 파일 I/O 경로는 잡지 못한다.

### 실제 발생 위치 (current version)
`encoding=` 누락 + `ensure_ascii=False`(비-ASCII 가능) 조합 — 트리거 가능:

| 파일:라인 | 코드 | 트리거 |
|----------|------|--------|
| `plugins/project-init/hooks/plan_gate_lib.py:210` | `tmp.write_text(json.dumps(state, ensure_ascii=False, indent=2))` | state 에 verifier_status `✅` → `/done`·`/approve-plan` 등 **모든 gate 전이 저장 실패** |
| `plugins/project-init/hooks/plan_gate_lib.py:1413` | `tmp.write_text(json.dumps(history, ensure_ascii=False, indent=2))` | history 에 한글/이모지 → 기록 실패 |
| `plugins/prompt-log/hooks/prompt_log_lib.py:124` | `tmp.write_text(json.dumps(allowed, ensure_ascii=False))` | 동의 목록에 비-ASCII 경로 → 쓰기 실패 |

> **불일치 증거**: 같은 파일의 `plan_gate_lib.py:558`(`audit_path.open("a", encoding="utf-8")`)·`:787`·`prompt_log_lib.py:256·411`(`os.fdopen(..., encoding="utf-8")`) 는 **이미 utf-8 명시**.
> 일부 경로만 고쳐져 있어 인코딩 정책이 파일 내 불일치 상태다. → 누락 라인만 맞추면 됨.

### 근본 수정 (범용 — 특정 문자/인코딩 하드코딩 금지)
1. 레포 전체 `write_text(`·`open(...,"w"/"a")` 를 grep 으로 전수 점검 → 텍스트 쓰기 전부 `encoding="utf-8"` 명시 (위 누락 3곳 포함).
   ```python
   tmp.write_text(json.dumps(state, ensure_ascii=False, indent=2), encoding="utf-8")
   ```
2. **일괄 안전망**: 위 라인별 수정과 별개로, 환경변수 `PYTHONUTF8=1` 은 Python UTF-8 모드를 켜
   **텍스트 I/O 전체(stdout + 파일) 기본을 UTF-8** 로 바꿔 §1 을 한 번에 덮는다.
   (DB 프로젝트에서 `/done` 을 `PYTHONUTF8=1` 로 실행해 통과시킨 실측 근거 있음)

### 임시 회피 (프로젝트측, 소스 무수정)
설치된 프로젝트 `.claude/settings.json` 최상위에:
```json
"env": { "PYTHONUTF8": "1" }
```
→ Claude Code 가 모든 하위 프로세스(훅·CLI)에 상속 → 파일쓰기 cp949 즉시 해소.
**주의**: 이건 *설치된* 프로젝트에만 즉효. 소스(plugin cache) 의 라인별 결함은 그대로이므로 신규 프로젝트엔 재발 → 근본 수정 필요.

## 2. 【중간】 settings.json 템플릿의 `python3` — 일부 Windows 환경에서 훅 전체 파손

### 현상
템플릿 `.claude/settings.json` 은 훅을 **`python3`** 로 호출한다.
Windows 에서 `python3` 가 **Microsoft Store 별칭 스텁**에 가로채인 머신(파이썬은 `Python3XX\python.exe` 로 설치)에서는
`python3` 가 스텁(무동작/스토어 안내)으로 풀려 **project-init·harness-update 직후 훅 5종이 전부 무동작**.

### 트리거 조건
- Windows + Python 을 python.org installer 로 설치(= `python3.exe` 미생성) + `WindowsApps` 별칭 활성.
- 이 DB 프로젝트가 정확히 이 케이스였고, 회피로 settings.json 의 모든 훅을 `Python312\python.exe` **절대경로**로 배선함.

### 근본 수정 (범용)
템플릿이 인터프리터를 하드코딩하지 않고 탐색하도록:
- 우선순위 탐색 래퍼: `py -3`(Windows 런처) → `python3` → `python` 중 **실제 동작하는 첫 번째**.
  Windows 는 `py` 런처가 스텁 문제를 우회하는 표준 진입점.
- 또는 SessionStart 부트스트랩에서 동작 인터프리터를 1회 감지해 settings 에 기록.
- 최소 조치: 템플릿 주석에 "python3 가 스텁에 가로채이면 절대경로/`py -3` 로 교체" 안내 1줄.

> harness-update 는 이 파일을 **사용자 선택 스킵** 처리한다(절대경로·enabledPlugins 보존). 즉 *업데이트* 는 안전하나, **신규 init** 는 위 머신에서 깨진 채 생성된다 — 근본은 템플릿 수정이어야 함.

## 3. 【낮음】 stdio reconfigure 에 `errors=` 미지정 — ✅ 해결됨 (§0 참조)

> **해결**: 전 엔트리포인트 가드를 `reconfigure(encoding="utf-8", errors="replace")` 로 통일(32 훅).
> `[17]` 가드의 substring 매칭(`reconfigure(encoding="utf-8")`)이 깨지지 않도록 닫는 괄호를 뗀
> `reconfigure(encoding="utf-8"` 로 같은 PR 에서 동시 수정했다. 아래는 원 분석 기록.

### 현상
§0 에서 삽입된 가드는 `_s.reconfigure(encoding="utf-8")` 로 **errors 핸들러 미지정**.
- 출력(유효 유니코드)은 안전.
- 단 **stdin 으로 cp949/깨진 바이트가 들어오면** utf-8 디코딩이 `UnicodeDecodeError` 로 여전히 죽을 수 있다(엣지).

### 근본 수정
가드를 `reconfigure(encoding="utf-8", errors="replace")` 로 통일 → 어떤 입력 바이트든 `�` 대체, crash 0.
(가이드 §1 의 범용 스니펫이 이미 `errors="replace"` 채택 — 소스 가드도 같은 정책으로 맞추면 됨.)

## 4. 【낮음】 pre-commit 의 lint 명령 PATH 의존 — CLAUDE.md 내용에 종속

### 현상
템플릿 `.githooks/pre-commit` 은 CLAUDE.md 명령어 섹션에서 lint 명령을 grep 으로 추출해 `eval` 한다(이전 `ruff check .` 하드코딩보다 개선됨).
그러나 CLAUDE.md 에 `ruff check .` 같이 **bare 명령**이 적혀 있으면 그대로 실행 → ruff 가 `.venv` 전용 설치인 환경에서 `command not found` 로 커밋 실패. 검사 범위(`.` 전체 vs staged) 도 CLAUDE.md 문구에 종속.

### 근본 수정 / 회피
- 프로젝트측: CLAUDE.md lint 명령을 venv 탐색형으로 기재(`python -m ruff check` 또는 `.venv/Scripts/ruff`), staged 한정은 명령에 포함.
- 또는 pre-commit 가 `python -m <linter>` 우선 + staged 파일만 전달하도록 보강.
- 임시: 커밋 시 `PATH="$PWD/.venv/Scripts:$PATH" git commit`.

> 카테고리상 §1·§2 와 같은 "환경 의존" 군이지만 현재는 **설정(CLAUDE.md) 종속** 으로 완화됨 — 심각도 낮음.

## 5. 【낮음·선택】 PyYAML 옵션 의존

`import yaml` 사용처(`cleanup_suggest.py:73`·`verifier_sandbox.py:52`·`validate_arch.py:39`·`pre-commit:66`·`prompt_log_lib.py:210`)는 **모두 try/except 폴백 보유** → 기능 장애 아님.
§0 의 stdout 수정으로 폴백 경고(한글)의 cp949 crash 도 사라졌다. **추가 조치 불필요**.
(선택) 의존 제거를 원하면 `constraints.yaml` 의 단순 구조를 `tomllib`(3.11+ stdlib)/JSON 으로 이관 가능.
