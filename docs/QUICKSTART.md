## 평소 흐름 — 이 3개면 끝

```
   ① /approve-plan   →   ② @verifier   →   ③ /done
      계획 승인             독립 검증            완료
```

1. **`tasks/todo.md` 에 뭘 할지 적고 → `/approve-plan`**
   - 계획을 승인하면 구현이 시작된다. (Claude 가 "승인해 주세요" 하고 먼저 안내해 준다)
2. **구현이 끝나면 → `@verifier`**
   - 독립 검증 에이전트가 **실제로 실행해서** 통과(✅)/실패(❌)를 판정한다.
3. **✅ 면 → `/done`**
   - 끝. 체크포인트가 정리되고 게이트가 닫힌다.

> 이 세 개만 기억하면 된다. 나머지 명령은 **막히거나 되돌리고 싶을 때만** 쓰고,
> 그 순간 Claude 가 "이걸 입력하세요" 하고 직접 알려준다 — 미리 외울 필요 없다.

## 막혔을 때 — 뭘 입력하지? (치트시트)

| 상황 | 입력 |
|------|------|
| 계획 다 짰고 시작하려는데 막혔다 | `/approve-plan` |
| 검증 ❌ — 고쳐서 다시 해볼래 | `/retry` |
| 검증 ❌ — 문제 알지만 일단 두고 마무리 | `/skip` |
| 지금까지 한 거 **전부 엎고** 시작점으로 | `/rollback` |
| 계획을 처음부터 다시 짤래 | `/replan` |
| 검증 없이 그냥 끝낼래 (판정 전에) | `/skip-verify` |
| 지금 상태가 어떤지 보고 싶다 | `/status` |
| plan-gate 잠깐 꺼둘래 | `/plan-gate-off` → 작업 → `/plan-gate-on` |

> **편집이 갑자기 막혔다면** 둘 중 하나다 — ① 같은 파일을 너무 여러 번(5회) 고침
> → 테스트를 한 번 통과시키거나 `/replan`. ② 검증 ❌ 후 잠김 → 위 치트시트에서 선택.

## 검증(@verifier)은 어떻게 보나

- 코드를 다 짠 뒤 `@verifier` 를 부른다. (Claude 가 자동으로 부르라고 환기하기도 한다)
- 결과가 **✅ → `/done`**, **❌ → `/retry`** 가 기본이다.
- ✅ 는 "**실제로 실행해서** 통과한 것"만 인정된다 — 코드를 읽기만 하고 통과시키면
  자동으로 ❌ 로 내려간다. 그러니 ✅ 가 뜨면 *진짜 돌려봤다*는 뜻이라 믿어도 된다.

---

## 파일 범위를 못박고 싶을 때 (선택 — 안 써도 됨)

"이번 작업은 `src/auth/` 만 건드려야 한다" 처럼 범위를 정하고 싶을 때만 쓴다.

- `tasks/todo.md` 에 범위를 선언하면 **기본은 "알림(shadow)"** 모드 — 범위를 벗어나면
  *막지는 않고 알려만* 준다.
- 진짜로 **막고** 싶으면 `/plan-gate-scope-enforce` (범위 밖 편집 거부 + Bash 가 만든
  범위 밖 새 파일 삭제). 게이트가 닫히면 자동으로 다시 "알림" 모드로 돌아온다.
- 완전히 끄려면 `/plan-gate-scope-off`.

> 대부분의 작업은 이걸 안 건드려도 된다. 범위를 선언하지 않으면 아무 일도 안 일어난다.

## 자주 묻는 것

**Q. `@verifier` 부르라는 안내가 자꾸 뜬다.**
검증을 아직 안 했다는 뜻. `@verifier` 를 부르거나, 검증이 필요 없으면 `/skip-verify`.

**Q. 되돌렸는데 내가 직접 만든 파일이 사라질까 봐 걱정된다.**
`/rollback` 은 Claude 가 편집한 파일만 되돌린다. **네가 직접 만든 파일은 건드리지 않는다.**

**Q. 비밀 파일(.env 등)을 읽으려는데 막힌다.**
의도된 보호다. `.env.example` 같은 예시는 허용된다.

**Q. 더 자세히 알고 싶다.**
상태 기계, 훅 전체 지도, 스코프 글롭 문법, 체크포인트·롤백 내부 동작 등은
[MANUAL.md](MANUAL.md) 에 전부 있다.
